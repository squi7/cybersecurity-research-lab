## Project / File

---

## Materi yang Dipelajari

### 1. Weak Sanitization (The "Satpam" Filter)

Mempelajari bagaimana fungsi `replace()` sering disalahgunakan untuk keamanan:

```javascript
let filteredInput = input.replace("<script>", "");
```

**Kelemahan:** - Hanya menghapus satu *occurrence* (kejadian pertama).

- *Case-sensitive* (tidak mendeteksi `<SCRIPT>`).
- Tidak menghapus tag alternatif (event handlers).

### 2. Bypass Skenario 1: Rekursif / Double Tag

Jika filter hanya menghapus `<script>` satu kali, payload bisa disusun agar ketika satu bagian dihapus, sisanya membentuk tag utuh kembali.

- **Payload:** `<scr<script>ipt>alert(1)</scr<script>ipt>`

### 3. Bypass Skenario 2: Event Handlers (Alternative Sinks)

Karena filter hanya fokus pada tag `<script>`, atribut HTML lain seperti `onerror` atau `onload` tetap lolos.

- **Payload:** `<img src=x onerror=alert(1)>` atau `<svg onload=alert(1)>`

---

## Sink Berbahaya Hari Ini

- `innerHTML` (Masih menjadi titik masuk utama eksekusi script di DOM).

---

## Cara Fix Aman

### 1. Gunakan Native API yang Aman

Ganti penggunaan `innerHTML` dengan:

```javascript
document.getElementById('search-result').textContent = filteredInput;
```

### 2. Library Sanitasi (DOMPurify)

Jika memang *harus* merender HTML, gunakan library yang sudah teruji daripada membuat filter `replace()` manual.

```javascript
let clean = DOMPurify.sanitize(input);
document.getElementById('search-result').innerHTML = clean;
```

---

## Pelajaran Penting

1. **Blacklisting itu sulit:** Mencoba memblokir kata kunci tertentu (seperti `<script>`) hampir selalu bisa di-bypass.
2. **Context Matters:** Filter yang dibuat tidak mempertimbangkan event handlers pada tag lain (seperti `<img>` atau `<body>`).
3. **Logic Flaw:** Fungsi `.replace()` di JavaScript secara default hanya mengganti elemen pertama yang ditemukan jika tidak menggunakan *Regular Expression* global (`/g`).
